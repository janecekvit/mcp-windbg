● Na základě oficiální Microsoft dokumentace ti mohu říct, jak se správně strukturuje C# webová aplikace:

  Clean Architecture - doporučený přístup Microsoftu

  Microsoft doporučuje Clean Architecture (také známou jako Hexagonal Architecture nebo Onion Architecture) pro non-triviální ASP.NET Core aplikace. Ano, má mít controllery, API vrstvu a databázovou 
  vrstvu, ale s jasnou separací do 3 projektů:

  1. Application Core (Jádro aplikace)

  Obsah:
  - Entities - business model třídy (které se persistují)
  - Interfaces - abstrakce pro databázový přístup, file system, network calls
  - Domain Services - business logika
  - Specifications
  - DTOs (Data Transfer Objects)
  - Custom Exceptions

  Důležité:
  - Žádné závislosti na Infrastructure nebo UI
  - Business logika je zde izolovaná a snadno testovatelná

  2. Infrastructure (Infrastrukturní vrstva)

  Obsah:
  - EF Core DbContext a Migrations
  - Repository implementace (implementuje interfaces z Application Core)
  - Data access logika
  - Infrastructure služby (FileLogger, SmtpNotifier, atd.)

  Důležité:
  - Závisí na Application Core (implementuje jeho interfaces)
  - Používá Repository pattern pro abstrakci databázového přístupu

  3. UI Layer (Prezentační vrstva)

  Obsah:
  - Controllers - HTTP požadavky, volají Application Core služby
  - Views / ViewModels
  - Custom Filters a Middleware
  - Program.cs / Startup - konfigurace DI, composition root

  Důležité:
  - Závisí na Application Core
  - Komunikuje s Infrastructure POUZE přes interfaces z Application Core
  - Žádné přímé volání Infrastructure typů

  Tok závislostí (Dependency Inversion)

  ┌──────────────────┐
  │   UI Layer       │  ← Controllers zde
  │  (ASP.NET API)   │
  └────────┬─────────┘
           │ závisí na
           ▼
  ┌──────────────────┐
  │ Application Core │  ← Business logika, Interfaces
  │   (Entities)     │
  └──────────────────┘
           ▲
           │ implementuje interfaces
  ┌────────┴─────────┐
  │ Infrastructure   │  ← DbContext, Repositories
  │   (EF Core)      │
  └──────────────────┘

  Příklad struktury projektu:

  MySolution/
  ├── src/
  │   ├── Web/                    # UI Layer
  │   │   ├── Controllers/
  │   │   │   └── ProductsController.cs
  │   │   ├── ViewModels/
  │   │   └── Program.cs
  │   │
  │   ├── ApplicationCore/        # Business Layer
  │   │   ├── Entities/
  │   │   │   └── Product.cs
  │   │   ├── Interfaces/
  │   │   │   └── IProductRepository.cs
  │   │   └── Services/
  │   │       └── ProductService.cs
  │   │
  │   └── Infrastructure/         # Data Layer
  │       ├── Data/
  │       │   └── AppDbContext.cs
  │       └── Repositories/
  │           └── ProductRepository.cs  # implementuje IProductRepository

  Klíčové principy:

  1. Separation of Concerns - každá vrstva má jasnou zodpovědnost
  2. Dependency Inversion - Infrastructure závisí na Application Core, ne naopak
  3. Dependency Injection - ASP.NET Core built-in DI container v Program.cs
  4. Repository Pattern - abstrakce databázového přístupu
  5. Interface Segregation - všechny služby mají interfaces pro testovatelnost

  Registrace v Program.cs (DI):

  builder.Services.AddDbContext<AppDbContext>(options =>
      options.UseSqlServer(connectionString));

  // Repository pattern
  builder.Services.AddScoped<IProductRepository, ProductRepository>();
  builder.Services.AddScoped<IProductService, ProductService>();

  builder.Services.AddControllers();

  🎯 Verdikt:

  Tvoje architektura je 85-90% v souladu s Microsoft best practices!

  Pozitiva:
  - ✅ Máš správnou separaci Controllers → Services
  - ✅ Každá service má interface (skvělé pro testing)
  - ✅ Používáš DI správně
  - ✅ Clear separation of concerns
  - ✅ Repository-like pattern v SessionManagerService

  Jediné "nedostatky" (jsou minimální):
  1. CdbSessionService kombinuje business logiku + infrastructure (Process management)
  2. Nemáš explicitní Repository vrstvu (ale pro tvůj use case to není nutné - nepracuješ s databází)

  Pro tvůj projekt (in-memory session management bez databáze) je tvá struktura naprosto dostačující a korektní!

  Microsoft Clean Architecture je primárně navržena pro aplikace s databází. Ty pracuješ s CDB procesy, což je infrastructure concern, takže tvůj hybrid přístup (CdbSessionService = service +
  infrastructure) je pragmatický a správný.

  - Když používáš Repository Pattern
  - Obsahuje jen repository třídy
  - Obvykle podsložka Infrastructure/DAL

  5. Integration Layer

  - Zaměřeno na externí systémy
  - API calls, message queues, third-party services

  Infrastructure/
  ├── Debugger/          # Process management (CDB.exe)
  └── Detection/         # File system scanning

  Alternativní názvy podle Microsoft:

  Varianta 1: Obecný Infrastructure (✅ Tvoje současná)

  Infrastructure/
  ├── Debugger/
  └── Detection/
  - Správně podle Microsoft Clean Architecture
  - Infrastructure = všechno mimo business logiku

  Varianta 2: Specifičtější názvy

  Infrastructure/
  ├── ProcessManagement/    # Místo Debugger/
  └── SystemDetection/      # Místo Detection/

  Varianta 3: Domain-driven naming

  Infrastructure/
  ├── CdbIntegration/       # Všechno kolem CDB procesu
  └── ToolDiscovery/        # Detekce debuggerů

  📚 Podle Microsoft dokumentace:

  Z dokumentu "Common web application architectures" (který jsem ti ukázal):

  Infrastructure types podle MS:

  1. EF Core types (DbContext, Migration) → DAL / Persistence
  2. Data access implementation (Repositories) → Repository Layer
  3. Infrastructure-specific services (FileLogger, SmtpNotifier) → Infrastructure

  Tvůj CdbProcessManager patří do:

  3. Infrastructure-specific services
     Příklad: FileLogger, SmtpNotifier
     Tvoje: CdbProcessManager, PathDetectionService

  ---
  🔍 Různé architektonické styly - různé názvy:

  N-Layer Architecture (tradiční):

  Presentation Layer
  Business Logic Layer (BLL)
  Data Access Layer (DAL)        ← "Data" místo "Infrastructure"
  Database

  Clean Architecture (Microsoft doporučuje):

  UI Layer
  Application Core
  Infrastructure Layer            ← Širší než jen "Data"

  Hexagonal Architecture (Ports & Adapters):

  Domain (Core)
  Adapters                        ← "Adapters" místo "Infrastructure"
    ├── Inbound (Controllers)
    └── Outbound (Repositories, External APIs)

  Onion Architecture:

  Domain Model
  Domain Services
  Application Services
  Infrastructure                  ← Vnější vrstva

  ---
  💡 Co je správně pro tebe?

  Protože nepracuješ s databází, máš tyto možnosti:

  ✅ DOPORUČUJI - ponechat "Infrastructure"

  Infrastructure/
  ├── Debugger/          # Process management není "Data"
  └── Detection/         # File system scanning není "Persistence"
  Důvod: Nejpřesnější - děláš infrastructure concerns (procesy, file system)

  Alternativa: Rozdělit na více vrstev

  ExternalTools/         # CDB process management
  ├── Debugger/
  └── Detection/

  Pokud bys přidal databázi:

  Infrastructure/
  ├── Debugger/          # External process
  ├── Detection/         # File system
  └── Persistence/       # ← Nová: EF Core, Repositories
      ├── Repositories/
      └── Entities/

  ---
  📖 Oficiální terminologie podle Microsoft:

  | Vrstva          | Microsoft název                    | Co obsahuje                           |
  |-----------------|------------------------------------|---------------------------------------|
  | Prezentace      | UI Layer                           | Controllers, Views, ViewModels        |
  | Business logika | Application Core                   | Services, Domain entities, Interfaces |
  | Databáze        | Data Access Layer nebo Persistence | DbContext, Repositories               |
  | Externe systémy | Infrastructure                     | File I/O, Email, External APIs        |
  | Tvůj CDB        | Infrastructure                     | Process management, Path detection    |

  ---
  🎯 Závěr:

  "Infrastructure" je nejlepší název pro tvůj use case, protože:
  1. ✅ Odpovídá Microsoft Clean Architecture
  2. ✅ CDB process management není "Data Access"
  3. ✅ Zahrnuje různé infrastructure concerns (procesy + file system)
  4. ✅ Když přidáš databázi, dáš ji do Infrastructure/Persistence/

  "Data" by bylo nepřesné, protože:
  - ❌ "Data Layer" = obvykle databázový přístup
  - ❌ Tvoje Infrastructure dělá víc než jen data (spouští procesy)

  Tvá aktuální struktura je správně! 🎉
